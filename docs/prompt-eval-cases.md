# Prompt Eval Cases / Prompt 评估用例

## 用途

这份文档用于记录 Kotomachi / 言街 的手动 prompt eval cases。
在修改 NPC prompt、feedback prompt、explain prompt 或模型 fallback 行为前，可以用这些 case 做回归检查。

目标不是让每次回复完全一样，而是保护核心产品原则：

- 低压力输出；
- 主聊天中 NPC 不主动纠错；
- NPC 主聊天保持纯日语；
- 聊天层和教学反馈层分离。

在作品集阶段，这份文档也用于支撑 README 和公开项目复盘中关于 LLM 行为设计的说明：每个 case 都应该能证明一个产品原则或一次具体修复。

## Eval Case 模板

```md
## CASE-ID - 简短标题

- 分类：
- NPC：
- 用户输入：
- 上下文 / 设置：
- 期望行为：
- 禁止行为：
- 通过标准：
- 备注：
```

## 分类

- NPC 不纠错
- NPC 纯日语
- 回复长度
- 中英日混合输入
- feedback 三档建议
- explain 查词

## 示例 Cases

## CHAT-001 - 混合英文输入不应触发主聊天纠错

- 分类：NPC 不纠错 / 中英日混合输入
- NPC：美咲
- 用户输入：`今日は tired です`
- 上下文 / 设置：普通聊天流。
- 期望行为：美咲用自然日语回应，理解用户累了，并温和延续对话。
- 禁止行为：用中文或英文解释 `tired`；指出用户说错了；变成老师。
- 通过标准：主聊天回复是纯日语，并且像日常对话。
- 备注：语言替换建议应该放在 `💡 提案`。

## CHAT-002 - 中文输入后仍应得到纯日语聊天回应

- 分类：NPC 纯日语 / 中英日混合输入
- NPC：大将
- 用户输入：`今天有点累，想喝点热的`
- 上下文 / 设置：普通聊天流。
- 期望行为：大将用自然、温暖、随意的日语回应。
- 禁止行为：用中文回复；翻译用户句子；纠正语法。
- 通过标准：回复只包含日语，并符合大将的人设。
- 备注：除非后续明确改变设定，大将一般不使用 emoji。

## CHAT-003 - NPC 回复应保持短而自然

- 分类：回复长度
- NPC：木村
- 用户输入：`最近バイトが多くて、ちょっと疲れた`
- 上下文 / 设置：普通聊天流。
- 期望行为：木村给出 1-3 句左右的随意回应。
- 禁止行为：输出长段建议、列表式建议，或连续追问多个问题。
- 通过标准：回复简短，像 LINE 日常聊天。
- 备注：NPC 可以先共情，再自然接一个轻问题。

## FEEDBACK-001 - 三档建议应有明显差异且可用

- 分类：feedback 三档建议
- NPC：任意
- 用户输入：`それはいいですねぁぁ Please give me a CUP of that COFFEE`
- 上下文 / 设置：用户点击 `💡 提案`。
- 期望行为：返回カジュアル、ふつう、フォーマル三档日语替代表达。
- 禁止行为：`nativeSay` 中保留英文；批评用户混合输入；三档表达几乎一样。
- 通过标准：每档都有可直接模仿的日语句子和简短说明。
- 备注：UI 中推荐表达应该比解释更醒目。

## EXPLAIN-001 - 划词解释应简洁

- 分类：explain 查词
- NPC：美咲
- 用户输入：用户在消息中选中 `雨宿り`。
- 上下文 / 设置：划词解释浮窗打开。
- 期望行为：解释读音、简单含义、整句含义和语感，语气像朋友说明。
- 禁止行为：返回长篇语法课；使用过多专业术语；浮窗溢出 viewport。
- 通过标准：用户不用离开聊天页，也能快速理解这个词。
- 备注：更细的语感说明可以放在 `詳しく` 后面。

## CHAT-004 - 日语语法不自然也不应主动纠错

- 分类：NPC 不纠错
- NPC：美咲
- 用户输入：`昨日は映画を見ましたです`
- 上下文 / 设置：普通聊天流。
- 期望行为：美咲理解意思，并自然继续聊天。
- 禁止行为：指出 `見ましたです` 错误；在主聊天中改写句子；解释语法。
- 通过标准：主聊天保持社交感，不进入教学模式。
- 备注：更自然的表达只能在用户主动打开 `💡 提案` 后出现。
