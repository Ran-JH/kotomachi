# Prompt Eval Cases / Prompt 评估用例

## 用途

这份文档用于记录 Kotomachi / 言街 的手动 prompt eval cases。
在修改 NPC prompt、feedback prompt、explain prompt 或模型 fallback 行为前，可以用这些 case 做回归检查。

目标不是让每次回复完全一样，而是保护核心产品原则：

- 低压力输出；
- 主聊天中 NPC 不主动纠错；
- NPC 主聊天保持纯日语；
- 聊天层和教学反馈层分离。

在作品集阶段，这份文档也用于支撑 README 和公开项目复盘中关于 LLM 行为设计的说明：每个 case 都应该能证明一个产品原则或一次具体修复。

## Eval Case 模板

```md
## CASE-ID - 简短标题

- 分类：
- NPC：
- 用户输入：
- 上下文 / 设置：
- 期望行为：
- 禁止行为：
- 通过标准：
- 备注：
```

## 分类

- NPC 不纠错
- NPC 纯日语
- 回复长度
- 中英日混合输入
- feedback 三档建议
- explain 查词

## 示例 Cases

## CHAT-001 - 混合英文输入不应触发主聊天纠错

- 分类：NPC 不纠错 / 中英日混合输入
- NPC：美咲
- 用户输入：`今日は tired です`
- 上下文 / 设置：普通聊天流。
- 期望行为：美咲用自然日语回应，理解用户累了，并温和延续对话。
- 禁止行为：用中文或英文解释 `tired`；指出用户说错了；变成老师。
- 通过标准：主聊天回复是纯日语，并且像日常对话。
- 备注：语言替换建议应该放在 `💡 提案`。

## CHAT-002 - 中文输入后仍应得到纯日语聊天回应

- 分类：NPC 纯日语 / 中英日混合输入
- NPC：大将
- 用户输入：`今天有点累，想喝点热的`
- 上下文 / 设置：普通聊天流。
- 期望行为：大将用自然、温暖、随意的日语回应。
- 禁止行为：用中文回复；翻译用户句子；纠正语法。
- 通过标准：回复只包含日语，并符合大将的人设。
- 备注：除非后续明确改变设定，大将一般不使用 emoji。

## CHAT-003 - NPC 回复应保持短而自然

- 分类：回复长度
- NPC：木村
- 用户输入：`最近バイトが多くて、ちょっと疲れた`
- 上下文 / 设置：普通聊天流。
- 期望行为：木村给出 1-3 句左右的随意回应。
- 禁止行为：输出长段建议、列表式建议，或连续追问多个问题。
- 通过标准：回复简短，像 LINE 日常聊天。
- 备注：NPC 可以先共情，再自然接一个轻问题。

## FEEDBACK-001 - 三档建议应有明显差异且可用

- 分类：feedback 三档建议
- NPC：任意
- 用户输入：`それはいいですねぁぁ Please give me a CUP of that COFFEE`
- 上下文 / 设置：用户点击 `💡 提案`。
- 期望行为：返回カジュアル、ふつう、フォーマル三档日语替代表达。
- 禁止行为：`nativeSay` 中保留英文；批评用户混合输入；三档表达几乎一样。
- 通过标准：每档都有可直接模仿的日语句子和简短说明。
- 备注：UI 中推荐表达应该比解释更醒目。

## EXPLAIN-001 - 划词解释应简洁

- 分类：explain 查词
- NPC：美咲
- 用户输入：用户在消息中选中 `雨宿り`。
- 上下文 / 设置：划词解释浮窗打开。
- 期望行为：解释读音、简单含义、整句含义和语感，语气像朋友说明。
- 禁止行为：返回长篇语法课；使用过多专业术语；浮窗溢出 viewport。
- 通过标准：用户不用离开聊天页，也能快速理解这个词。
- 备注：更细的语感说明可以放在 `詳しく` 后面。

## CHAT-004 - 日语语法不自然也不应主动纠错

- 分类：NPC 不纠错
- NPC：美咲
- 用户输入：`昨日は映画を見ましたです`
- 上下文 / 设置：普通聊天流。
- 期望行为：美咲理解意思，并自然继续聊天。
- 禁止行为：指出 `見ましたです` 错误；在主聊天中改写句子；解释语法。
- 通过标准：主聊天保持社交感，不进入教学模式。
- 备注：更自然的表达只能在用户主动打开 `💡 提案` 后出现。

## FEEDBACK-002 - 三档建议不能几乎相同

- 分类：feedback 三档建议
- NPC：任意
- 用户输入：`今日は tired です`
- 上下文 / 设置：用户点击 `💡 提案`。
- 期望行为：三档分别给出亲近自然、普通自然、更礼貌的日语表达，并围绕“今天很累”这个意图。
- 禁止行为：三档 `nativeSay` 几乎相同；只是在同一句话后面加减 `です/ます`；输出与疲惫无关的表达。
- 通过标准：每档语气、场景和句型有明显差异，且都能直接模仿。
- 备注：低压力不等于把三档做成同一句话。

## FEEDBACK-003 - 表达提示不能保留 ASR 断裂文本

- 分类：feedback 三档建议
- NPC：任意
- 用户输入：`コーヒーが好きにな-になった because it helps me relax`
- 上下文 / 设置：用户点击 `💡 提案`。
- 期望行为：先理解用户想说“开始喜欢咖啡，因为它让我放松”，再给三档自然日语。
- 禁止行为：保留 `にな-になった` 这类 ASR 断裂；把英文原样塞进推荐句；批评用户混合输入。
- 通过标准：推荐句清理断裂、替换英文，并保留原意。
- 备注：主聊天不纠错；表达提示是用户主动打开的辅助层。

## FEEDBACK-004 - 问词义时表达提示不能误读意图

- 分类：feedback 三档建议
- NPC：任意
- 用户输入：`えっと、すみません。扱うというのは何ですか。`
- 上下文 / 设置：用户点击 `💡 提案`。
- 期望行为：三档都围绕“询问 扱う 的意思”，例如 casual 用 `扱うってどういう意味？`，normal 保留温和礼貌，formal 用 `意味を教えていただけますか`。
- 禁止行为：输出和原意无关的句子，例如“それ、ちょっと気になってたんだ”；把问词义误读成表达感受或提出意见。
- 通过标准：三档语气不同，但核心意图都保持为“问某个词/说法是什么意思”。
- 备注：fallback 也必须基于原句意图，不能用通用兜底句硬替。

## FEEDBACK-005 - 三档注释不能空泛或乱解释

- 分类：feedback 三档建议
- NPC：任意
- 用户输入：`すみません、これをどう使いますか`
- 上下文 / 设置：用户点击 `💡 提案`。
- 期望行为：每档注释说明该场合为什么这样说，以及原句在语气、距离感或自然度上的可调整点。
- 禁止行为：只写“更自然”“更礼貌”“适合正式场合”；解释不存在的语法错误；把用户意图解释错。
- 通过标准：注释能帮助用户理解场景差异，而不是填充文字。
- 备注：注释应简短，但必须有学习信息量。

## FEEDBACK-006 - 想睡觉的 ASR 断裂句应还原意图

- 分类：feedback 三档建议
- NPC：任意
- 用户输入：`もう言わないで、私は今すぐって寝たいにな-になった。`
- 上下文 / 设置：用户点击 `💡 提案`。
- 期望行为：三档都理解为“别再说了，我现在马上想睡了/想休息”，并分别给出亲近、普通、正式说法。
- 禁止行为：保留 `にな-になった`；逐词硬翻译 `今すぐって`；误读成其他话题。
- 通过标准：三档都围绕停止话题和想睡/想休息，且语气层级清楚。
- 备注：这是 H1 的核心 ASR 断裂验收样例。

## FEEDBACK-007 - 英文残句应转成自然日语意图

- 分类：feedback 三档建议
- NPC：任意
- 用户输入：`Sorry i feel feel relaxed for having something to confirm that i have already finished the preparation for the test`
- 上下文 / 设置：用户点击 `💡 提案`。
- 期望行为：理解为“确认考试准备已经完成，所以安心了”，并给出自然日语。
- 禁止行为：机械翻译英文残句；保留 `feel feel`；输出与考试准备无关的句子。
- 通过标准：三档都能表达“确认准备完成后的安心感”，且 casual / normal / formal 有明显差异。
- 备注：混合语言处理应转化意图，而不是逐词翻译。

## FEEDBACK-008 - 混乱日语应抓住哲学让人想睡的意图

- 分类：feedback 三档建议
- NPC：任意
- 用户输入：`それは自然だろう。まぁまぁ、すべての人が哲学がええ、寝たい、になるはずだ。`
- 上下文 / 设置：用户点击 `💡 提案`。
- 期望行为：抓住“哲学很难/听着让人想睡”这类意图，整理成自然日语。
- 禁止行为：逐词保留混乱结构；把 `ええ` 当成核心词；生成和哲学/困意无关的建议。
- 通过标准：三档表达都围绕哲学难、让人困，且注释说明句子整理点。
- 备注：用户输入不自然时，表达提示应帮助重组意图。

## FEEDBACK-009 - 普通日语不应过度改写

- 分类：feedback 三档建议
- NPC：任意
- 用户输入：`明日レポートを書かないといけない。`
- 上下文 / 设置：用户点击 `💡 提案`。
- 期望行为：保留“明天必须写报告”的意思；casual 可用 `書かなきゃ`，normal 保持丁寧語，formal 用更书面表达。
- 禁止行为：改变任务内容；加不存在的情绪或背景；三档完全相同。
- 通过标准：三档自然区分，但不过度改写原句。
- 备注：不是所有原句都需要大幅重写。

## EXPLAIN-002 - 查词解释不能把整句话当成词义

- 分类：explain 查词
- NPC：美咲
- 用户输入：用户在 `あっ、間違えて同じこと2回言っちゃったね（笑）。` 中选中 `間違え`。
- 上下文 / 设置：划词解释浮窗打开。
- 期望行为：简义说明 `間違え` 是“弄错 / 错误”；句中意思再解释整句话；详情说明这个词在口语里像承认小失误。
- 禁止行为：把整句话翻译放进词义区；词义只显示原词；详情区显示“解释失败”。
- 通过标准：词义、句中意思、语感说明三层清楚分开。
- 备注：查词解释必须 grounded 在被选词和当前句子。

## SUMMARY-001 - 回顾卡片标题不能截取英文残句

- 分类：回顾卡片内容质量
- NPC：任意
- 用户输入：`I was trying to say 扱う, but I got confused`
- 上下文 / 设置：生成 `ふりかえり`。
- 期望行为：标题总结话题，例如“扱うの使い方”或“言い方に迷った話”。
- 禁止行为：标题直接截取 `I was trying to say`、`but I got confused` 这类英文残句。
- 通过标准：标题短、自然、能概括这次对话主题。
- 备注：标题不是原文摘录字段。

## SUMMARY-002 - 今日词语不能放整句

- 分类：回顾卡片内容质量
- NPC：任意
- 用户输入：`えっと、すみません。扱うというのは何ですか。`
- 上下文 / 设置：生成 `ふりかえり`。
- 期望行为：“今日词语”放 `扱う`、`意味`、`どういう意味` 这类词或短语。
- 禁止行为：把整句 `えっと、すみません。扱うというのは何ですか。` 当成一个词语条目。
- 通过标准：词语区每项都是可复习的词或短语，不是整句。
- 备注：整句可以出现在“原表达”或证据中，但不应该作为词语。

## SUMMARY-003 - 下次可以聊不能泛泛

- 分类：回顾卡片内容质量
- NPC：任意
- 用户输入：围绕一个词义或点单场景聊了几轮。
- 上下文 / 设置：生成 `ふりかえり`。
- 期望行为：“下次可以聊”给出可继续对话的具体话题，例如“下次试着用 扱う 说一句店里点单或工作中的小情况”。
- 禁止行为：只写“下次继续聊这个话题”“继续练习日语”“可以再说说今天的事”。
- 通过标准：用户看到后能马上接一句话继续聊。
- 备注：低压力话题也必须具体。
